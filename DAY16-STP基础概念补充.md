# STP 基础概念补充（Day 16）

## 一、为什么需要 STP？

在复杂网络中，设备之间会建立多条物理链路来实现**冗余备份**。但冗余链路必然带来**环路**风险，环路会导致：

- **广播风暴**：广播帧在环路中不断增生、循环，耗尽带宽和 CPU
- **MAC 地址漂移**：交换机从多个端口学到同一 MAC，导致 MAC 表震荡

STP（生成树协议）通过**逻辑上阻塞冗余端口**来打破环路，同时保留物理链路的备份能力。


## 二、BPDU 的核心比较原则

STP 按照以下顺序选择最优的配置 BPDU：

1. **最小的根桥 ID**
2. **最小的 RPC（根路径开销）**
3. **最小的网桥 ID（发送者桥 ID）**
4. **最小的接口 ID（发送端口 ID）**

> 四步逐级比较，前一步相同才进入下一步。


## 三、配置 BPDU 的四个核心参数

| 参数        | 含义                                     | 关键点                                                |
| ----------- | ---------------------------------------- | ----------------------------------------------------- |
| **根桥 ID** | 根桥的身份证（优先级 + MAC）             | 全网唯一不变，确认谁是根桥                            |
| **RPC**     | 从发送端口到根桥的累计路径开销           | 判断哪个口离根桥最近，**越小越优**                    |
| **网桥 ID** | 发送 BPDU 的交换机身份证（优先级 + MAC） | RPC 相同继续比，**小者胜**；区别于根桥 ID（全网统一） |
| **接口 ID** | 发送 BPDU 的端口编号（优先级 + 端口号）  | 前三项相同最后兜底，**小者胜**                        |

**特别说明**：选举根接口时，比较的是该 BPDU 中携带的**根桥侧端口 ID**（即根桥发出该 BPDU 的端口），而非本地接收端口 ID。

**比较顺序口诀**：

> 根桥 ID → RPC → 网桥 ID → 接口 ID


## 四、路径开销（Cost）参考

**带宽越大，开销越小**，因为带宽高说明路径质量更好。

| 链路带宽 | STP 开销（802.1t 标准） |
| :------: | :---------------------: |
| 10 Mbps  | 200,000（旧标准为 100） |
| 100 Mbps |  19,000（旧标准为 19）  |
|  1 Gbps  |   4,000（旧标准为 4）   |
| 10 Gbps  |   2,000（旧标准为 2）   |

> **速记**：带宽越大 → 开销数值越小。


## 五、STP 选举流程（三步走）

### 第 1 步：选举根桥（Root Bridge）

- 比较**桥 ID**（优先级 → MAC），**最小者**胜出
- 本拓扑：SW1 胜出（MAC 最小）

### 第 2 步：选举根接口（Root Port）

- 每台**非根桥交换机**有且仅有 1 个根接口
- 比较**RPC**（到根桥开销），**最小者**胜出
- 本拓扑：SW2 Port1、SW3 Port1 胜出（直连 100M，RPC = 19）

### 第 3 步：选举指定接口（Designated Port）

- **每条链路**有且仅有 1 个指定接口
- 比较顺序：**RPC → 网桥 ID → 端口 ID**，**最小者**胜出
- 本拓扑：SW2 Port2 胜出（与 SW3 RPC 相等，但桥 ID 更小）

### 第 4 步：剩余端口阻塞

- 本拓扑：SW3 Port2 阻塞，环路打破

### 拓扑结构示意

```
SW1（根桥）
├── 100M（Cost=19）── SW2 Port1（RP）
│                     └── 10M（Cost=200000）── SW3 Port2（阻塞）
│
└── 100M（Cost=19）── SW3 Port1（RP）
```

### 最终端口角色清单

| 设备 | 端口角色                           |
| ---- | ---------------------------------- |
| SW1  | 双端口均为 DP（根桥默认）          |
| SW2  | Port1 = RP，Port2 = DP             |
| SW3  | Port1 = RP，Port2 = 阻塞（Non-DP） |

### 核心口诀

> **根桥**看 MAC 最小（桥 ID 最小）
> **RP** 看 RPC 最小
> **DP** 看完整四步（RPC → 桥 ID → 端口 ID）
> **剩下端口**全阻塞

### 注意事项

- RPC 相同时**不能直接下结论**，必须继续比桥 ID
- 阻塞口**仍接收 BPDU**，只是不转发用户数据
- 根桥**没有根接口**，所有端口均为指定接口


## 六、STP 端口角色

| 角色       | 全称                        | 数量规则                    | 功能                                     | 最终状态 |
| ---------- | --------------------------- | --------------------------- | ---------------------------------------- | -------- |
| **RP**     | 根端口（Root Port）         | 每台非根桥**有且仅有 1 个** | 去往根桥的最佳入口，接收 BPDU 和业务数据 | 转发     |
| **DP**     | 指定端口（Designated Port） | 每条链路**有且仅有 1 个**   | 将 BPDU 和业务数据从根桥方向转发到该链路 | 转发     |
| **Non-DP** | 非指定端口                  | 链路上除 DP 外的剩余端口    | 被阻塞以打破环路，不转发业务数据         | 阻塞     |

**一句话总结**：

> 每台非根桥选 1 个 RP（上行入口），每条链路选 1 个 DP（下行出口），剩下的全部阻塞。

**补充说明**：

- 根桥所有端口都是 DP
- 端口角色由 STP 计算决定，端口状态是计算后的结果
- 只有 RP 和 DP 才能进入转发状态


## 七、STP 端口状态

### 五种状态详解

| 状态                   | 收发 BPDU | 收发业务数据  | 学习 MAC | 说明                                          |
| ---------------------- | :-------: | :-----------: | :------: | --------------------------------------------- |
| **禁用（Disabled）**   |     ✗     |       ✗       |    ✗     | 端口物理 Down 或手动关闭，完全不工作          |
| **阻塞（Blocking）**   | ✓（仅收） |       ✗       |    ✗     | 被 STP 阻塞打破环路，持续侦听 BPDU            |
| **侦听（Listening）**  |     ✓     |       ✗       |    ✗     | 被选为 RP/DP 但计算未结束，持续 15 秒         |
| **学习（Learning）**   |     ✓     | ✗（只收不转） |    ✓     | 为转发做准备，学 MAC 但不转发数据，持续 15 秒 |
| **转发（Forwarding）** |     ✓     |       ✓       |    ✓     | 正常工作，只有 RP 或 DP 可进入此状态          |

### 状态迁移条件

| 迁移        | 触发条件                                               |
| ----------- | ------------------------------------------------------ |
| 阻塞 → 侦听 | 端口被选为 RP 或 DP；或阻塞超时（20 秒）               |
| 侦听 → 学习 | 等待 1 个 Forward Delay（15 秒），确保拓扑稳定         |
| 学习 → 转发 | 再等待 1 个 Forward Delay（15 秒），确保无环路         |
| 转发 → 阻塞 | 拓扑变化，端口角色重新计算后不再是 RP/DP               |
| 阻塞 → 转发 | 持续 20 秒收不到 BPDU → 侦听 15 秒 → 学习 15 秒 → 转发 |

### 补充说明

- 阻塞口**仍收 BPDU**（用于感知拓扑变化），禁用口**完全不工作**
- 学习状态收数据是为了**学 MAC**，不是正式转发
- **只有 RP 和 DP** 能进入转发状态，Non-DP 永远停在阻塞状态


## 八、STP 计时器

### 三个关键计时器

| 计时器            | 默认值 | 作用                                  |
| ----------------- | ------ | ------------------------------------- |
| **Hello Time**    | 2 秒   | 根桥发送配置 BPDU 的间隔              |
| **Forward Delay** | 15 秒  | 端口在侦听或学习状态停留的时间        |
| **Max Age**       | 20 秒  | BPDU 的老化时间，超时未收到则重新计算 |

### 收敛时间场景对比

| 场景                   | 耗时         | 过程                                                   |
| ---------------------- | ------------ | ------------------------------------------------------ |
| **根桥整体失效**       | **约 50 秒** | 阻塞口等 20 秒老化 → 侦听 15 秒 → 学习 15 秒 → 转发    |
| **根端口故障**         | **约 30 秒** | 阻塞口已有最新 BPDU，直接 15 秒侦听 + 15 秒学习 → 转发 |
| **边缘端口（接终端）** | **0 秒**     | 默认转发，不经任何状态过渡                             |

**速记公式**：

> **50 秒 = 20 秒老化 + 15 秒侦听 + 15 秒学习**

### 边缘端口（Edge Port）补充

- **特性**：默认状态即为转发，0 秒上线，不经过阻塞 → 侦听 → 学习 → 转发流程
- **安全机制**：
  - 若边缘端口收到 BPDU（说明有人接了交换机），会**立即丧失边缘端口特性**，变回普通端口，重新参与 STP 计算（经历完整五状态）
  - 若开启 **BPDU 保护**，收到 BPDU 会直接 **error-down** 该端口，并通知管理员


## 九、华为默认计时器汇总

| 计时器        | 默认值 |
| ------------- | ------ |
| Hello Time    | 2 秒   |
| Forward Delay | 15 秒  |
| Max Age       | 20 秒  |


## 十、关键区别速查

| 对比项   | 根桥 ID      | 网桥 ID         |
| -------- | ------------ | --------------- |
| 属于谁   | 根桥         | BPDU 发送者自身 |
| 是否变化 | 全网统一固定 | 每台设备不同    |
| 比较时机 | 第一步比较   | RPC 相同后比较  |
